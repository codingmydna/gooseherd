### 1. "Choosing a Claude model and effort level in Claude Code" тАФ Claude Code team, ~2026-07-08
https://claude.com/blog/claude-model-and-effort-level-in-claude-code

**Pattern:** model and effort are two independent dials, with a diagnostic heuristic for which one to raise: "did Claude not know enough, or not try hard enough?" тАФ not enough context/capability тЖТ bump model; skipped steps, bailed early, didn't run tests тЖТ bump effort. On genuinely hard multi-step work, a bigger model can reach the quality bar in fewer iterations than a smaller model at max effort, sometimes at lower total cost per task, and can solve things the smaller model can't at any effort level.

**Parity impact:** gooseherd already exposes the mechanism (`GOOSE_<ROLE>_MODEL` / `GOOSE_<ROLE>_EFFORT` per role тАФ planner/implementer/reviewer/judge/evaluator), but has never written down this decision framework. Added as a new ЁЯФ╢ row in section 4 тАФ doc-only gap.

## Pick of the week

**Gap:** gooseherd has no automated, judgment-based permission approval outside of `/orch`'s narrow, deterministic implementer allowlist. Both upstreams are actively investing here right now: Claude Code just made classifier-based Auto mode the *default* recommendation on three more platforms (2026-07-11), and Codex just shipped a more granular declared-read/write approval mode (2026-07-09). A plain interactive `goose` session still has only "prompt every time" or the existing implement-allowlist rules тАФ nothing in between.

**Ready-to-run `/orch` task:**

> Add a lightweight auto-judge permission mode for interactive `goose` sessions. Introduce `GOOSE_SESSION_AUTO_JUDGE_MODEL` / `GOOSE_SESSION_AUTO_JUDGE_EFFORT` (unset by default тАФ current prompt-every-time behavior is unchanged). When set, route each `RequestPermissionRequest` through a cheap judge role that classifies the request as low-risk (read-only, workspace-scoped write, or an already-allowlisted command) vs. high-risk (network access, destructive operation, out-of-workspace write, or an unlisted command) before falling back to the existing prompt/allowlist paths тАФ never override an explicit deny. Reuse the existing tool-kind classification in `crates/goose/src/acp/provider.rs` (`permission_decision_from_mode`, `clean_command_allowlist`) rather than reimplementing it. Done when: at least 12 unit tests cover the read/write/execute/network/destructive classification matrix (mocked judge responses), the existing orch-implement-allowlist and Plan-Explore-policy tests show zero regressions, and `cargo fmt`, `cargo clippy --workspace --all-targets -- -D warnings`, and `cargo test -p goose` all pass.
